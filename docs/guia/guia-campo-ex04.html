ex04 — SDK Bridge

La cumbre. Dejas de usar Claude Code y empiezas a integrarlo: tu programa en C++ lo llama, lo acota y consume sus resultados. El círculo del módulo, cerrado con tus manos.

requiere: TODO el módulo anterior ~4-6 h de trabajo real aquí no hay soluciones

Por qué este ejercicio es la cumbre

Hasta ahora Claude Code era una herramienta que usabas. A partir de hoy es una capacidad que tus programas tienen. Esa frontera — de usuario a integrador — es la que separa «sé usar IA» de «sé construir con IA». Y el subject tiene razón: si entendiste cada pieza anterior, el puzzle se monta solo. Si no... se nota aquí.

WP·01

Rumbo — qué vas a aprender y por qué importa

El Agent SDK permite invocar a Claude Code desde código, programáticamente: tu programa lanza una consulta, Claude trabaja, y te devuelve el resultado. El SDK es TypeScript/Python — y tu programa es C++. ¿Problema? Ninguno: es el pan de cada día de la programación de sistemas. Cuando dos mundos no hablan el mismo idioma, construyes un puente: un proceso intermedio y un formato de intercambio.

Tu puente tendrá tres tramos: el programa C++ (CodeReviewer) lanza con system() un script Node (bridge.mjs); el script usa el SDK para pedirle a Claude que analice un archivo; y el resultado vuelve como JSON, que tu C++ parsea y muestra. Lenguajes distintos, un contrato común. Como una calculadora RPN conectada a un teodolito: cada aparato con su firmware, y un protocolo en medio que los entiende a ambos.

La idea que lo sostiene todo

El JSON intermedio es el contrato del puente — la interfaz estable entre dos mundos que no se conocen. Si te suena al .hpp del ex00, no es casualidad: este módulo lleva desde el primer día enseñándote la misma idea con distintos disfraces. Definir bien las fronteras es controlar el sistema.

WP·02

Terreno — los conceptos, desde cero

El Agent SDK — Claude Code como librería
En lugar de teclear en la CLI, tu código llama a la función query(): «analiza este archivo y devuélveme los issues». Mismo motor, distinta puerta de entrada. Se instala con npm (@anthropic-ai/claude-code) y se usa desde JavaScript/TypeScript — de ahí el bridge. Un matiz que pilla a todos: query() no devuelve una respuesta única, sino un flujo de mensajes que recorres hasta quedarte con el de tipo result.
Autenticación — tu sesión, no una API key
query() usa la misma sesión de Claude Code en la que ya iniciaste sesión en ex00–ex03. No necesitas ANTHROPIC_API_KEY ni tarjeta: el análisis corre con tu suscripción, no por la API de pago por token. Es el Claude Code de siempre, ahora invocado desde tu código — por eso este ejercicio no te cuesta un céntimo si hiciste los anteriores.
allowedTools — el mínimo privilegio
El SDK te deja acotar qué herramientas puede usar Claude. ["Read"] significa: puedes leer, y nada más — ni escribir, ni ejecutar bash. Para un revisor de código es exactamente lo que quieres: que mire, no que toque. Es el principio de mínimo privilegio de toda la vida, y es la versión programática del control que llevas construyendo todo el módulo.
system() — el puente rápido (y sus peajes)
Lanza un comando via shell y espera a que termine. Simple y suficiente para el mandatory. Sus pegas: pasa por la shell (cuidado con qué interpolas en la cadena), y te da poco control sobre el proceso hijo. Por eso el bonus propone fork()/exec() — más código, más control, y una demostración de que sabes lo que hay debajo.
El JSON como contrato
sample_report.json define el formato: archivo, lista de issues (línea, tipo, mensaje), resumen. Tu C++ debe parsearlo y tu bridge debe generarlo — idéntico. Mientras ambos respeten el contrato, puedes cambiar cualquiera de los dos lados sin romper el otro. Eso es una interfaz bien hecha.
El matiz que separa a un junior de alguien que controla

Los tres errores que el subject te obliga a gestionar (fichero no encontrado, la llamada a query() que falla —sesión caída o sin login—, JSON malformado) no son relleno: son las tres formas en que el puente se cae, una por tramo. Quien los gestiona bien demuestra que entiende la arquitectura; quien pone un try/catch genérico demuestra que copió el patrón.

WP·03

Ruta — cómo atacar el ejercicio

El subject ya te da los pasos incrementales — y son sagrados. La estrategia de esta ruta es una sola palabra: aislar. Cada tramo del puente se construye y se prueba por separado, antes de unirlo al siguiente.

  1. Paso 1: C++ a solas con el JSON de ejemplo.

    Sin SDK, sin Node, sin Claude. Solo tu programa leyendo sample_report.json y mostrándolo legible. Decide aquí tu estrategia de parseo y sé honesto con sus límites en NOTES.md: ¿parser propio acotado al formato del contrato? Defendible, si sabes decir qué casos no cubre.

  2. Paso 2: el bridge a solas, desde la terminal.

    node bridge.mjs test_files/leak.cpp a mano, sin C++ por medio. ¿Genera un JSON con el formato del contrato? ¿Respeta allowedTools: ["Read"]? No avances hasta que este tramo aguante solo — es la mejor inversión de tiempo del ejercicio.

  3. Paso 3: unir las orillas.

    Ahora sí: system() desde CodeReviewer, leer el JSON generado, mostrar. Y los tres errores del subject, provocados a propósito: renombra el fichero, rompe el JSON a mano, simula el SDK caído. Si no has visto fallar cada caso, no sabes si lo gestionas.

  4. Paso 4: la validación con los tres sospechosos.

    leak.cpp, uninit.cpp, logic_bug.cpp — cada uno esconde su delito. Ejecuta, verifica que Claude los detecta, y documenta el resultado. Si alguno se le escapa, eso también va al NOTES.md: un revisor con falsos negativos conocidos vale más que uno con fe ciega.

  5. El bonus final: cierra el círculo.

    Hook Stop (ex02) que lanza el review (ex04) y te manda el resumen a Telegram. Es el momento «lanzas el proceso, te vas al CrossFit, y el resultado te llega al móvil» — y la demostración viva de que el módulo entero era UN sistema, no cinco ejercicios sueltos.

# La regla de oro de esta ruta: cada tramo se prueba AISLADO
$ ./reviewer sample_report.json # tramo C++ — ¿parsea?
$ node bridge.mjs test_files/leak.cpp # tramo SDK — ¿genera?
$ ./reviewer test_files/leak.cpp # puente completo — ¿aguanta?
WP·04

Zonas de riesgo — donde se atasca la gente

!

Saltarse el paso 1 «porque es fácil». Lo es. Por eso existe: cuando el flujo completo falle (y fallará), necesitas saber que el tramo C++ era sólido para descartar capas. Quien se salta el paso fácil acaba depurando tres tramos a la vez — la definición técnica de una tarde perdida.

!

Parsear JSON con find() y fe. Buscar "line": a pelo por el string funciona... hasta que un mensaje de Claude contiene esa subcadena. Si haces parser propio, acótalo al contrato y documenta los límites; si esa fragilidad te incomoda (bien), razona la alternativa en NOTES.md. Lo indefendible es no haberlo pensado.

!

Dejar allowedTools abierto. Sin la lista, Claude podría escribir archivos o ejecutar comandos durante un simple análisis. ¿Probabilidad de desastre? Baja. ¿Razón para asumirla? Cero. Un revisor lee. Mínimo privilegio no es paranoia: es diseño.

!

Confiar en que el SDK siempre responde bien. Red caída, sesión expirada, output truncado: tu C++ recibirá basura alguna vez, y un parser confiado se la come con un crash feo. Valida antes de parsear. Es el paso 3 del subject, no un extra para perfeccionistas.

!

El token de Telegram en el repo público. El clásico que duele: haces el bonus, commiteas, publicas... con el token del bot dentro. Variables de entorno o archivo ignorado por git, siempre. Que tu primer proyecto público de IA no incluya tu primera fuga de credenciales.

WP·05

Checkpoint — la defensa de la cumbre

En la evaluación de este ejercicio te pueden pedir que ejecutes tu CodeReviewer sobre un archivo que no has visto jamás. Prepárate con esto:

  1. Dibuja (literalmente, en papel) el flujo completo: C++ → system() → bridge → SDK → Claude → JSON → C++. ¿En qué tramo vive cada uno de los tres errores que gestionas?
  2. ¿Por qué el JSON intermedio hace que puedas reescribir el bridge en Python mañana sin tocar una línea de C++?
  3. ¿Qué riesgos concretos tiene system() y qué ganas exactamente con fork()/exec()? ¿Cuándo NO compensaría el cambio?
  4. ¿Por qué allowedTools: ["Read"] y no la lista completa? Conecta tu respuesta con lo que aprendiste en el ex02.
  5. Te doy un .cpp nuevo y tu reviewer no detecta el bug que contiene. ¿Qué tres hipótesis compruebas, y en qué orden?

Si puedes responder la Q1 dibujando sin dudar, has hecho cumbre. Date la vuelta y mira el valle: /init, contexto, hooks, MCP... todo el camino era esto. ¡Por Tutatis!
