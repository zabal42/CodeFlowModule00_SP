ex03 — PlayBot

Hasta ahora tu agente vivía encerrado en tu repo. Hoy le das un navegador. MCP es la puerta; Playwright, las manos. Y el patrón que aprendes vale para cualquier puerta.

requiere: ex00–ex02 · node ≥ 18 ~2-3 h de trabajo real aquí no hay soluciones
📄 Leer el Subject (PDF)

Una de ética antes de soltar al bot

El subject es tajante y yo lo subrayo: solo webs públicas. Nada de cuentas con credenciales, nada de contenido privado, nada de scraping masivo. Un navegador automatizado es una herramienta seria — úsala como alguien que quiere que le contraten, no como alguien que quiere salir en las noticias.

WP·01

Rumbo — qué vas a aprender y por qué importa

MCP (Model Context Protocol) es el sistema de plugins de Claude Code: un protocolo estándar para que el agente hable con herramientas externas. Si vienes del mundo C/Unix, la analogía buena es POSIX: da igual quién implemente el sistema operativo — si cumple el estándar, tu código funciona encima. Con MCP igual: da igual quién implemente el servidor (Playwright, un filesystem, GitHub...), Claude habla con todos en el mismo idioma.

Y aquí está la clave del ejercicio, que el subject te dice a la cara: Playwright es solo un ejemplo. Lo que de verdad te llevas es el patrón universal — configurar, verificar, usar — que repetirás con cualquier servidor MCP que conectes el resto de tu vida profesional. El navegador es la excusa divertida; el patrón es la lección.

La idea que lo sostiene todo

Un agente es tan capaz como las herramientas que le conectas — y tan peligroso como las herramientas que le conectas sin pensar. MCP te da el poder de ampliar; este ejercicio te enseña a ampliarlo con cabeza.

WP·02

Terreno — los conceptos, desde cero

La arquitectura: cliente y servidor
Claude Code es el cliente MCP. Un servidor MCP es un proceso aparte que expone herramientas («abrir página», «hacer click», «leer elemento»...) siguiendo el protocolo. Claude no sabe nada de navegadores: sabe que tiene unas herramientas nuevas disponibles y para qué dice cada una que sirve. La inteligencia está en el cliente; las manos, en el servidor.
.mcp.json — la declaración
Aquí declaras qué servidores existen y cómo arrancarlos (el comando que lanza el proceso). Es como el fstab de tus herramientas: lo que no esté montado ahí, no existe para Claude.
La verificación — el paso que todos se saltan
Configurar no es conectar. Antes de pedir nada, comprueba que Claude Code ve el servidor y lista sus herramientas (el comando /mcp te enseña el estado de cada servidor conectado). Este paso intermedio es el que separa «funciona» de «creo que funciona», y el subject te pide pantallazo de él por una buena razón.
Playwright — las manos en el navegador
Un framework de automatización de navegadores: abre páginas, localiza elementos, hace clicks, lee contenido. Via MCP, Claude decide qué hacer y Playwright lo ejecuta. Tú escribes el objetivo; la pareja se reparte el trabajo.
El matiz que separa a un junior de alguien que controla

Cada acción del navegador mete contexto: el contenido de las páginas, los resultados de cada herramienta... Todo va a la ventana que aprendiste a vigilar en el ex01. Un agente navegando alegremente es una manguera de tokens — acota la tarea, pide datos concretos, y cierra cuando termines.

WP·03

Ruta — cómo atacar el ejercicio

  1. Instala y declara el servidor.

    Servidor MCP de Playwright instalado y registrado: con claude mcp add --scope project playwright npx @playwright/mcp@latest o editando el .mcp.json de la raíz (el starter trae plantilla). Si Node no está a la altura (≥ 18), este es el momento de descubrirlo, no en mitad del test.

  2. Verifica ANTES de usar — y captura el momento.

    Arranca Claude Code y comprueba que lista las herramientas del servidor. Ese pantallazo es entregable obligatorio, pero sobre todo es tu seguro: si algo falla más adelante, ya sabes que la conexión estaba bien y el problema es otro.

  3. Extracción quirúrgica, no safari.

    Para el informe del perfil de GitHub, escribe un prompt de francotirador: qué datos exactos (nombre de repos, bio, lenguajes), de qué URL, a qué archivo (reports/github_profile.md). Cuanto más concreto el encargo, menos páginas visita el bot y menos contexto quema.

  4. Diseña el test antes de lanzarlo.

    Define TÚ el criterio: qué página, qué significa PASS (status 200, elemento X presente) y qué significa FAIL. Luego pide a Claude que lo ejecute y reporte con timestamp. Un test cuyo criterio de éxito se decide después de ver el resultado no es un test, es una coartada.

  5. El NOTES.md va del patrón, no de Playwright.

    El error típico es documentar «cómo usé Playwright». Lo que se evalúa es que entiendas el patrón MCP: qué declara la config, qué demuestra la verificación, cómo se usa una herramienta externa. Escríbelo de forma que sirviera igual para un servidor de filesystem — entonces lo has entendido.

# El triple salto del ejercicio
1. .mcp.json # configurar — declarar la puerta
2. /mcp # verificar — ¿la puerta existe y abre?
3. prompt concreto # usar — cruzarla con un objetivo claro
WP·04

Zonas de riesgo — donde se atasca la gente

!

Saltarse la verificación e ir directo al prompt. Si el servidor no cargó, Claude no te va a decir «no tengo Playwright»: intentará apañárselas con lo que tiene, fallará de formas creativas, y tú perderás una hora depurando el prompt cuando el problema era la fontanería. Verifica. Siempre. Primero.

!

El encargo vago. «Mírame el perfil de GitHub y haz un informe» es una invitación a que el bot pasee. Resultado: informe flojo y factura de tokens gorda. Datos concretos, URL concreta, archivo de destino concreto — el ex00 ya te enseñó que la precisión en las instrucciones no es opcional.

!

No saber en qué capa falla. Algo no funciona: ¿es el servidor que no arranca, el protocolo que no conecta, o Claude que no entiende el encargo? Son tres capas distintas con tres arreglos distintos. Aprende a aislarlas: ¿lista las herramientas? La conexión vive. ¿Las llama y devuelven error? Mira el servidor. ¿Ni las llama? Mira tu prompt.

!

Olvidar el taxímetro del ex01. Cada página que el bot visita es contexto que entra. Si tu sesión de extracción se alarga, ya sabes qué pasa con la calidad del razonamiento — y con la factura. Tarea acotada, sesión corta, y a otra cosa.

!

Cruzar la línea de lo público. «Es que con mi cuenta logueada el test era más fácil» — no. El subject lo prohíbe, el sentido común lo prohíbe, y tu futuro yo en una entrevista también. Webs públicas, datos públicos, punto.

WP·05

Checkpoint — ¿defenderías esto mañana?

  1. ¿Qué papel juega exactamente .mcp.json, y qué pasa con un servidor que no está declarado ahí?
  2. El informe sale vacío. Describe tu proceso para averiguar en qué capa está el fallo: servidor, conexión o prompt.
  3. ¿Por qué el patrón configurar-verificar-usar sirve igual para un servidor de filesystem que para Playwright? ¿Qué cambia y qué no?
  4. ¿Qué impacto tiene en la ventana de contexto una sesión de navegación larga, y qué dos medidas del ex01 aplicarías?
  5. Si tuvieras que conectar un segundo servidor MCP mañana (el bonus), ¿cuáles serían tus tres primeros pasos, en orden?

Si la Q3 te sale fluida, el ejercicio está aprendido — Playwright era el decorado, el patrón era la obra.
